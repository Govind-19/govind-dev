quick follow-up to [the multi-tenant story](/journal/every-school-thinks-its-the-only-one), because this is how that one actually bit me.

the rule was simple: every schema change updates *both* sources of truth тАФ the numbered migration (for existing schools) and the base schema (for new ones). and the way i "enforced" it was.... i'd ask claude every time, "did you update both?" ЁЯд╖тАНтЩВя╕П

it worked. for about two or three changes.

then i did what every human does with a manual checklist тАФ i stopped doing it. not on purpose. you're moving fast, the migration looks right, you merge. ask once, ask twice, and by the tenth change you've quietly decided claude has probably got it ЁЯЩГ

it did not always have it. and the base schema and the migrations started drifting apart. silently. for a long time.

<Pull>a check that depends on me remembering isn't a check. it works twice, then it's decoration.</Pull>

we only found out because a bug surfaced in the app тАФ and chasing it forced us to admit the two schema sources had genuinely diverged. so we pointed the claude agent at the whole codebase and had it diff them.

it came back with a pile. tables that didn't match. columns in one and not the other. months of small, un-mirrored changes ЁЯШотАНЁЯТи we went through and brought them back in line.

that could've been the end of it тАФ "okay, i'll remember next time". but i'd already proved i won't. the answer wasn't more discipline. it was making the check impossible to skip.

so i asked claude to build it for me. it landed as three small files:

```text
.claude/settings.json                  тЖТ registers the hook
.claude/hooks/check-migration-sync.py  тЖТ a python hook that runs on its own
.claude/commands/sync-tenant-schema.md тЖТ the /sync-tenant-schema command,
                                         a prompt, not a shell script
```

and the flow turned out more subtle than "a script notices and fixes it" тАФ which is honestly how i'd pictured it before i read what claude actually wired up:

1. **i edit a file.** it's a `PostToolUse` hook matching `Write|Edit`, so it fires right after *my* tool calls. edit a file by hand in your IDE and it stays silent тАФ only the agent's edits trip it.
2. **the harness runs `check-migration-sync.py`,** piping in JSON of what i just did, including the file path.
3. **the hook checks the path** for `database/migrations/tenant/`. not a tenant migration? it says nothing, we move on. is one? it prints a small JSON message.
4. **here's the part i had backwards** тАФ the hook does *not* run the sync, and it doesn't diff anything itself. all it does is drop a line into my context: *"you MUST now run /sync-tenant-schema."* (that's the `additionalContext` field.)
5. **i read that reminder** and choose to run `/sync-tenant-schema` myself.
6. **and that command is a prompt, not a bash script** тАФ running it just loads the audit steps from the `.md` into my context. then i do the real work with normal tools (`git diff`, `ls`, read, edit) тАФ checking whether `database/tenant-schema/tables/` drifted from the migration, reporting what's off, and asking before i change a thing.

so it's a nudge, not an autopilot. the hook can't fix the schema, can't even block the edit. all it guarantees is that i can't *not* notice тАФ and the actual fix still runs through a prompt that asks me first.

<Ex>Ex: i turned "did you update both?" from a question i had to remember to ask.... into a thing the repo asks for me, every single time, automatically.</Ex>

and there's a neat symmetry to it. the AI is what made it easy to drift тАФ change one file, move on. and the AI, pointed by a hook i set up, is now what catches the drift. the tool didn't save me. the guardrail i built around the tool did ЁЯд╖тАНтЩВя╕П

<Pull>don't trust yourself to remember the boring, critical check. encode it somewhere it can't be skipped.</Pull>
