quick follow-up to [the multi-tenant story](/journal/every-school-thinks-its-the-only-one), because this is how that one actually bit me.

the rule was simple: every schema change updates *both* sources of truth — the numbered migration (for existing schools) and the base schema (for new ones). and the way i "enforced" it was.... i'd ask claude every time, "did you update both?" 🤷‍♂️

it worked. for about two or three changes.

then i did what every human does with a manual checklist — i stopped doing it. not on purpose. you're moving fast, the migration looks right, you merge. ask once, ask twice, and by the tenth change you've quietly decided claude has probably got it 🙃

it did not always have it. and the base schema and the migrations started drifting apart. silently. for a long time.

<Pull>a check that depends on me remembering isn't a check. it works twice, then it's decoration.</Pull>

we only found out because a bug surfaced in the app — and chasing it forced us to admit the two schema sources had genuinely diverged. so we pointed the claude agent at the whole codebase and had it diff them.

it came back with a pile. tables that didn't match. columns in one and not the other. months of small, un-mirrored changes 😮‍💨 we went through and brought them back in line.

that could've been the end of it — "okay, i'll remember next time". but i'd already proved i won't. the answer wasn't more discipline. it was making the check impossible to skip.

so i asked claude to build a small **skill**, wired to a **hook**:

```text
hook: on every edit/write to a migration file
   │
   └─► run a script that diffs  base-schema  ⟷  versioned migrations
          │
          ├─ in sync?  → say nothing, carry on
          └─ drifted?  → hand the diff back to claude:
                         "these don't match. sync them."
```

the key bit — it doesn't silently auto-fix. the moment a migration file is touched, the script checks the two against each other, and if they've drifted it feeds that difference straight back to the agent as context, with the instruction to reconcile them. the verification i kept forgetting now happens on its own — the instant the drift is created, inside the same loop where i can still catch it.

<Ex>Ex: i turned "did you update both?" from a question i had to remember to ask.... into a thing the repo asks for me, every single time, automatically.</Ex>

and there's a neat symmetry to it. the AI is what made it easy to drift — change one file, move on. and the AI, pointed by a hook i set up, is now what catches the drift. the tool didn't save me. the guardrail i built around the tool did 🤷‍♂️

<Pull>don't trust yourself to remember the boring, critical check. encode it somewhere it can't be skipped.</Pull>
